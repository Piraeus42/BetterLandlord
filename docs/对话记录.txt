  调查结论：符号流控制流分析

  1. RNG 流的调用链（正常工作部分）

  在 add_cards() 中，每个卡片选择流程：

  稀有度判定:  _bh_rng_rarity.randf()          → 消费 1 个稀有度 RNG 值
  符号选取:    _bh_rng_sym_<rarity>.randi_max()  → 消费 1 个该稀有度符号 RNG 值
  池变更:      card_pool[rarity].erase(picked)   → 从池中移除已选符号（防止同批重复）

  RNG 消费量是确定的：同一批 3 张卡，稀有度 RNG 消费 3 次，各稀有度符号 RNG 按实际命中次数消费。种子相同 → RNG
  吐出相同数值序列。

  ---
  2. 🔴 核心问题：池突变 + 索引选择 → 序列被打乱

  你的理想模型：sym_rare 流是个固定序列 [A, B, C, D, ...]，每次需要稀有符号就从流里取下一个。

  但实际行为：RNG 给出的是数组索引，不是固定序列。而且数组（pool）在同一个 add_cards() 调用内会变化。

  具体示例：

  初始 card_pool["rare"] = ["diamond", "ruby", "emerald", "sapphire"]
                         索引:  0          1        2          3

  时间线 A — 卡片1选了 "emerald" (索引2)：
  卡1: rare → sym_rare.randi_max(4) → 值=2 → 选中 pool[2]="emerald"
       erase("emerald") → pool = ["diamond", "ruby", "sapphire"]
                                        索引:  0        1        2
  卡2: uncommon → 走 sym_uncommon 流（不影响 rare）
  卡3: rare → sym_rare.randi_max(3) → 值=1 → 选中 pool[1]="ruby"

  时间线 B — 卡片1选了 "diamond" (索引0)：
  卡1: rare → sym_rare.randi_max(4) → 值=2 → 选中 pool[2]="??? "

  等等，这里有一个更微妙的问题。RNG 给的第一个值是 2（固定的），但在时间线 B 中，如果卡1选的符号不同，那是因为
  randi_max(4) 不可能返回不同的值（RNG 流是确定的）...

  让我更仔细地重新建模：

  时间线 A：
  卡1: rarity RNG=0.15 → rare → sym_rare.randi_max(4)=2 → pool[2]="emerald"
       erase("emerald") → pool = ["diamond", "ruby", "sapphire"]
  卡2: rarity RNG=0.80 → uncommon → sym_uncommon picks something
  卡3: rarity RNG=0.25 → rare → sym_rare.randi_max(3)=1 → pool[1]="ruby"
  结果: 卡1="emerald", 卡3="ruby"

  时间线 B（卡1用户选了和A不同的稀有符号——比如"diamond"）：

  问题就在这里！如果卡1选的还是rare稀有度，那么 RNG 的第一个值 2 固定返回 pool[2]="emerald"。用户不可能在相同 RNG
  值下选不同的符号——除非用户跳过了卡1（点 Skip），改变了后续 RNG 调用次数。

  但如果卡1是 rare，卡2是 common（不同稀有度），然后卡3也是 rare：
  卡1: rarity RNG=0.15 → rare → sym_rare.randi_max(4)=2 → pool[2]="emerald"
       erase("emerald") → pool = ["diamond", "ruby", "sapphire"]
  卡2: rarity RNG=0.45 → common → sym_common picks something
  卡3: rarity RNG=0.25 → rare → sym_rare.randi_max(3)=1 → pool[1]="ruby"

  vs 如果卡1被 forced_rarity 改变了稀有度：
  卡1: forced_rarity → common → sym_common.randi_max picks something
  卡2: rarity RNG=0.15 → rare → sym_rare.randi_max(4)=2 → pool[2]="emerald"
       注意！这里 pool["rare"] 还没被擦除，所以大小仍是 4
  卡3: rarity RNG=0.45 → common → ...

  关键洞察： 因为池在一个 add_cards()
  调用内是可变的（已选符号被擦除），同一稀有度在同批中第二次命中时，池大小已经变了。相同的 RNG
  值除以不同的池大小，得到不同的余数 → 不同符号。

  而且由于擦除的元素不同（第一次选了什么），即使池大小一样，剩余元素的排列也不同，索引可能指向完全不同的元素。

  4. 总结：你设想的固定序列是否存在？

  不存在。 原因不是随机数被偷，而是索引模型（index-based）+ 可变池（mutable pool）。

  你的设计给每个稀有度分配了独立 RNG 流，这确保了 RNG 值的确定性。但选取逻辑是：
  symbol = card_pool[rarity][randi_max(card_pool[rarity].size())]

  而不是：
  symbol = fixed_sequence_for_rarity.pop_front()

  因为 card_pool[rarity] 在同批 3 张卡中会被 erase() 修改，同一个 RNG 值落到不同的池内容上就会产生不同结果。

  这个效应在同批 3 张卡内最显著（当 2 张卡共享同一稀有度时）。跨批次（下次 add_cards 调用）池会从 rarity_database 重新
  duplicate，初始状态一致，所以跨批次的 RNG 消费量一致，但被选中的符号集合不同 → 进入游戏状态的符号不同 → 后续
  can_add_highlander()、essences_unlocked 等条件改变 → 后续池的构建（erase highlander、erase essence_capsule）也不同 →
  逐批次累积发散。

  标准三选一（stcf <= 3）路径中，没有对符号池做 shuffle。代码是：

  card_pool = rarity_database["symbols"].duplicate(true)

  批次开始: card_pool["rare"] = [diamond, ruby, emerald, sapphire]  (原始顺序)

  ⑥ forced_rarity_arr.shuffle() 的影响

  这是唯一和符号选择共享 RNG 的 shuffle。它和稀有度判定共用 _bh_rng_rarity。

  正常流程（无 forced_rarity）:
    卡0: rarity.randf() → 判定稀有度
    卡1: rarity.randf() → 判定稀有度

    卡2: rarity.randf()

  次，导致同一种子下后续稀有度判定全部偏移

  3. Reel/Effect shuffle 只在每 spin 临时流上操作 — 不影响跨 spin 确定性

  最反直觉的发现

  StS2 不分稀有度隔离 RNG 流。它用的是 任务隔离（Rewards / Shops / Transformations / CombatCardGeneration...），而不是
  稀有度隔离。

  他们的 CombatCardGeneration RNG 处理所有稀有度的卡牌生成——先选稀有度，再在该稀有度内选卡，但都用同一个
  RNG。因为每张卡固定消费 2 次 RNG（1 次稀有度 + 1 次选择），序列天然确定。

  我们的"按稀有度分 RNG"反而创造了问题——当一批次的 common 数量波动时（3,2,3 vs 2,2,2），sym_common
  的消费量波动，映射就错位了。

  结论

  StS2 模式不能直接搬到幸运房东。

  ┌────────────────┬───────────────────────────────────────────────────┬────────────────────────────────────────────┐
  │                │                       StS2                        │                  幸运房东                  │
  ├────────────────┼───────────────────────────────────────────────────┼────────────────────────────────────────────┤
  │ 每批卡数       │ 3 张（固定）                                      │ 3 张（固定） ✓                             │
  ├────────────────┼───────────────────────────────────────────────────┼────────────────────────────────────────────┤
  │ 稀有度判定触发 │ 每次奖励必有（每局固定次数）                      │ 每次 add_cards 必有 ✓                      │
  ├────────────────┼───────────────────────────────────────────────────┼────────────────────────────────────────────┤
  │ 稀有度分池     │ ❌ 不分池，所有卡混在 CardPoolModel 里            │ ✅ 按稀有度分池                            │
  ├────────────────┼───────────────────────────────────────────────────┼────────────────────────────────────────────┤
  │ RNG 消费       │ 每卡固定 2 次，与稀有度无关                       │ 稀有度不同 → 分不同 sym_* RNG →            │
  │                │                                                   │ 消费量取决于稀有度频率                     │
  ├────────────────┼───────────────────────────────────────────────────┼────────────────────────────────────────────┤
  │ 池重复约束     │ blacklist 过滤，不 mutation                       │ erase 变异                                 │
  ├────────────────┼───────────────────────────────────────────────────┼────────────────────────────────────────────┤
  │ 核心前提       │ 每批次的 pool                                     │ 每个稀有度有独立 symbol pool               │
  │                │ 是全量过滤（所有稀有度的卡都在同一个池）          │                                            │
  └────────────────┴───────────────────────────────────────────────────┴────────────────────────────────────────────┘

  StS2 能 work 是因为它的 池不按稀有度分——所有卡都在 CardPoolModel 里，RollForRarity 后只是 filter。RNG
  消费不因稀有度而异。

  我们的池是按稀有度桶分的，这个核心差异决定了不能照搬。贸然统一 RNG 反而会把问题搞得更糟——同一个 RNG
  值会被路由到不同的稀有度池。
